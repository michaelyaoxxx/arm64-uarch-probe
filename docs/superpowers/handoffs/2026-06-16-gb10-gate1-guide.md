# GB10 Gate 1 远程协作测试指导

> **角色分工:** 本文档写给 **GB10 硬件操作者**。Mac 端开发者在 `codex/phase3-implementation`
> 分支上完成了 Phase 3 全部实现（349 测试通过），现在由操作者在 GB10 真机上执行 Gate 1
> 并将结果传回。

## 1. 背景：Gate 1 是什么

Phase 3 实现了统一测量 Runner 和三个标准化探针（`chase_pmu`、`evict_slc`、
`chase_migrate`）。Gate 1 是 **最小的可控冒烟测试**：在 GB10 真机上跑
`--profile smoke`（仅 2 个 case），验证全链路——编译、计划、执行、环境恢复、
结果持久化——在真实硬件上正常工作。

**Gate 1 不做**：大规模测量、resume/rerun 测试、性能分析、基线采集。
这些属于 Phase 4。

## 2. GB10 机器前提条件

| 条件 | 检查方式 | 备注 |
|------|----------|------|
| ARM64 Linux (GB10 SoC) | `uname -m` → `aarch64` | 必须 |
| `git` 可用 | `git --version` | 从 GitHub clone |
| CPython 3.13.13 | `python3.13 -V` | 没有的话 `uv` 会自动处理 |
| `uv` 可用 | `uv --version` | 安装: `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| C 编译器 (`cc`) | `cc --version` | 编译 C 探针 |
| `root` 或等效权限 | `sudo -v` 或已是 root | `--allow-mutation` 需要 |
| SSH + GitHub 访问 | `ssh -T git@github.com` | clone + push 结果 |

### 快速自检脚本

```sh
uname -m                # 必须是 aarch64
git --version
uv --version || echo "请安装 uv: curl -LsSf https://astral.sh/uv/install.sh | sh"
cc --version
sudo -v || echo "需要 sudo 权限"
ssh -T git@github.com   # 应该看到 "Hi <user>! You've successfully authenticated..."
```

## 5. 执行流程（对照 runbook 9 步）

### Step 1: 记录 Commit 和干净树证据

```sh
git rev-parse HEAD > results/gate1-commit.txt
git status --short >> results/gate1-commit.txt
cat results/gate1-commit.txt
```

**预期:** `git status --short` 无输出（干净树）。

### Step 2: 捕获工具链证据

```sh
mkdir -p results
uv run --no-sync python -V > results/gate1-toolchain.txt
uv --version >> results/gate1-toolchain.txt
cc --version >> results/gate1-toolchain.txt
uname -srm >> results/gate1-toolchain.txt
cat results/gate1-toolchain.txt
```

> **注意:** `uv run --no-sync python -V` 第一次运行会自动下载 CPython 3.13.13，
> 这是正常的，等待完成即可。

### Step 3: 同步依赖并编译探针

```sh
make sync      # 创建 .venv/，安装 CPython 3.13.13
make build     # 编译 C 探针到 build/bin/
```

验证编译产物：

```sh
file build/bin/chase_pmu > results/gate1-build.txt
file build/bin/evict_slc >> results/gate1-build.txt
file build/bin/chase_migrate >> results/gate1-build.txt
cat results/gate1-build.txt
```

**预期:** 三个文件都报告为 ELF 可执行文件。

### Step 4: 运行 Phase 3 验收测试

```sh
make phase3-check 2>&1 | tee results/gate1-phase3-check.txt
```

**预期:** 所有测试通过，末尾输出 `OK`。

> 如果测试失败：记录完整输出到 `results/gate1-phase3-check-FAILED.txt`，
> 不要继续后续步骤，直接跳到第 7 节"结果回传"。

### Step 5: 运行 Doctor（执行前环境快照）

```sh
./probe doctor -o json > results/gate1-doctor.json
cat results/gate1-doctor.json
```

**预期:** 退出码 `0`；JSON 包含 `backend_id`、`observations`、`journals` 字段。
`journals` 应为空数组 `[]`（没有未完成的 journal）。

### Step 6: 生成 Plan

```sh
./probe plan --platform gb10 --profile smoke -o json > results/gate1-plan.json
cat results/gate1-plan.json
```

**预期:** 退出码 `0`；JSON 中包含 `cases`（2 个 case）和 `environment_phases`。

### Step 7: 执行 Smoke Run ★ 核心步骤

```sh
./probe run --platform gb10 --profile smoke --allow-mutation \
    --output-dir results/gate1-runs
```

**预期:** 退出码 `0`（所有 case `status: "ok"`）。

查找生成的 RunResult：

```sh
ls -la results/gate1-runs/
# 文件名类似: 20260616T120000Z-a1b2c3d4.json
echo results/gate1-runs/$(ls results/gate1-runs/ | head -1) > results/gate1-run-path.txt
cat results/gate1-run-path.txt
```

> 如果退出码为 `15`（探针执行失败）：不要惊慌。记录完整输出到
> `results/gate1-run-FAILED.txt`，继续执行 Step 8（检查环境是否恢复），
> 然后在第 7 节回传结果。不要在 GB10 上直接 debug 或修改代码。

### Step 8: 验证环境恢复

```sh
./probe doctor -o json > results/gate1-doctor-after.json
diff <(jq -S '.journals' results/gate1-doctor.json) \
     <(jq -S '.journals' results/gate1-doctor-after.json)
```

**预期:** `journals` 数组为空（`[]`），与执行前一致。`diff` 无输出。

### Step 9: 不要扩展

- ❌ **不要** 在 GB10 上执行 `probe resume`（AC5 已经在 Mac fixture 上验证）
- ❌ **不要** 跑 `--profile full` 或其他 scenario
- ❌ **不要** 修改代码
- ❌ **不要** 在 GB10 上 debug 探针失败

## 7. 结果回传流程

### 方案 A：推送到新分支（推荐）

在 GB10 机器上：

```sh
# 1. 确保在仓库根目录
```

## 8. 故障排查

### 常见问题

| 症状 | 可能原因 | 解决 |
|------|----------|------|
| `make sync` 失败 | 网络不通，无法下载 CPython | 检查 `curl -I https://github.com` |
| `make build` 失败 | 缺少 `cc` 或编译依赖 | `cc --version`；检查 `make show-targets` |
| `make phase3-check` 失败 | 部分测试在 GB10 上行为不同 | 记录完整输出，不要跳过 |
| `probe run` 退出码 11 | 缺少 `--allow-mutation` 或权限不足 | 确认 `--allow-mutation` 已加；`sudo -v` |
| `probe run` 退出码 14 | 残留 journal 或锁文件 | `./probe doctor -o json` 检查 journals |
| `probe run` 退出码 15 | 探针执行失败 | 记录输出；这可能就是 Gate 1 要发现的问题 |
| `probe doctor` 有残留 journal | 上次运行异常中断 | `./probe restore --journal <path> --allow-mutation` |

### 紧急联系

如果上述步骤无法解决，在 `results/` 下创建一个 `ISSUES.txt` 文件：

```sh
cat > results/ISSUES.txt << 'EOF'
步骤: <Step N>
命令: <实际运行的命令>
退出码: <退出码>
现象: <描述看到的现象>
完整输出:
<粘贴终端输出>
EOF
```

将 `ISSUES.txt` 随结果一起传回。

## 9. 安全和注意事项

- **`--allow-mutation` 只加在 `probe run` 上**，不要在其他命令上加。
- **CLI 从不调用 `sudo`**——由操作者自行决定用 root shell 还是 `sudo` 前缀运行整个 `./probe` 命令。
- `make smoke` **不需要** `--allow-mutation`（它在 fixture 上跑），在 GB10 上可以跳过。
- Step 8 的环境恢复验证很重要——确认 `journals` 为空才说明 Transaction 系统在 GB10 上工作正常。
- 不要在 GB10 上修改代码。任何修改都应在 Mac 端完成，通过 PR 合并回 `main`，然后 GB10 重新 pull。
