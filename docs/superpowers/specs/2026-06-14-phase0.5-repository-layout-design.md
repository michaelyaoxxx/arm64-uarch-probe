# Phase 0.5 Repository Layout Design

## Purpose

Phase 0.5 makes the repository's ownership boundaries visible before the Phase
0 GitHub closure, without breaking frozen historical evidence or prematurely
implementing Phase 1 interfaces.

The repository currently mixes historical experiment assets with emerging
v1.0 infrastructure at the top level. This phase documents which paths are
frozen, transitional, or owned by v1.0, then adds trackable directory skeletons
for future work.

## Directory Classes

### Frozen Historical Evidence

- `runner/run_pmu*.sh`
- `data/`

These paths remain in place and continue to be protected by
`legacy/manifest.json`. New v1.0 features must not modify them or add new files
under them. Moving them requires a separately reviewed compatibility migration
that preserves traceability.

### Transitional Paths

- `analysis/`
- `baseline/`
- `runner/cache_info_*.sh`

These paths remain unchanged during Phase 0.5. Phase 2 or Phase 3 will decide
the ownership of cache-information helpers. Phase 4 will import historical
analysis and baseline material into the structured result protocol before any
archive or migration decision.

### v1.0-Owned Paths

Phase 0.5 creates these trackable skeletons:

```text
arm64_probe/
configs/
  platforms/
  experiments/
  profiles/
tests/
  unit/
  contract/
  fixtures/
  integration/
docs/
  methodology/
  references/
  results/
  roadmap/
```

Each skeleton directory contains a concise `README.md` defining its purpose,
allowed content, and ownership boundary. README files are used instead of
`.gitkeep` files or Python `__init__.py` files so Phase 0.5 does not lock in
Phase 1 package or configuration interfaces.

## Authoritative Layout Contract

Create `docs/design/repository-layout.md` as the authoritative contributor-facing
layout contract. It must:

- classify current and future paths as frozen, transitional, or v1.0-owned;
- explain why frozen paths remain at their historical locations;
- identify the phase that may resolve each transitional path;
- state that directory movement uses `git mv` where practical;
- prohibit new v1.0 features in frozen paths;
- link to the legacy integrity and result-retention policies.

Update `AGENTS.md`, `README.md`, and `docs/design/repository-contract.md` only as
needed to point contributors to the authoritative layout contract.

## Contract Tests

Extend repository policy tests to verify:

- every v1.0 skeleton directory has a tracked `README.md`;
- the layout contract names every frozen, transitional, and v1.0-owned path;
- entry documents point to `docs/design/repository-layout.md`;
- the canonical legacy manifest continues to contain only historical
  `runner/run_pmu*.sh` and `data/**/*.txt` paths;
- no frozen runner or data file changes during Phase 0.5.

The tests enforce repository structure without asserting future Python package
names, module APIs, configuration encodings, or backend implementations.

## Non-Goals

Phase 0.5 does not:

- move `runner/run_pmu*.sh` into `legacy/runner/`;
- move or rewrite `data/`;
- migrate `analysis/`, `baseline/`, or cache-information helpers;
- create Python package markers, domain models, CLI behavior, or configuration
  schemas;
- require Linux ARM64 or GB10 execution.

## Acceptance

Phase 0.5 is complete when the layout contract and all skeleton README files are
committed, repository tests pass on Mac, the legacy manifest still verifies,
and the diff contains no changes under frozen or transitional paths.
